Convention 01

Fields

Standard business fields — things like CustomerName or InvoiceDate — carry no prefix. The zz prefix system is reserved exclusively for organizational, calculated, key, metadata, helper, and summary fields, pushing them below business fields when the field list is sorted alphabetically.

Field Name Structure

Anatomy
zzT__FieldName_suffix

zz        →  Pushes field to bottom when sorted alphabetically
T         →  Single-character type code  (see table below)
__        →  Visual separator between prefix and name
FieldName →  PascalCase descriptive name
_suffix   →  Optional type hint  (see suffix table below)

Type Codes

PrefixTypeDescription
zzc__CalculationComputed values derived from other fields. Never edited directly.
zzg__GlobalSingle value shared across all records. Used for filters, parameters, and UI state.
zzk__KeyPrimary and foreign key fields. The key type (pk / fk) sits between the prefix and name so foreign keys group together, with the primary key sorting last.
zzm__Metadata / AuditRecord-level audit fields — creator, modifier, timestamps.
zzr__Relationship HelperFields that exist solely to support a relationship or portal filter. No standalone business meaning.
zzs__SummaryAggregate values such as totals, counts, or averages across a found set.

Optional Suffixes

SuffixTypeDescription
_tsTimestampA date-and-time value. Common on audit fields.
_dtDateA date-only value where time is irrelevant.
_flagBooleanA binary true/false switch (typically 0 or 1).
_idKey ReferenceA non-key field that holds a key value — e.g. a global for portal filtering. Not used on actual key fields.
_pkPrimary KeyMarks the primary key. Placed between prefix and name: zzk__pk_RecordID. Only used with zzk__.
_fkForeign KeyMarks a foreign key. Placed between prefix and name: zzk__fk_CustomerID. Only used with zzk__.
_cntCountA numeric field representing a count.
_r[n]RepeatingA repeating field with n repetitions. Always placed last when combining: zzg__SearchDate_dt_r10.
💡
Key vs. Reference zzk__fk_CustomerID is the actual foreign key field. zzg__SelectedCustomer_id is a global that holds a customer key value. pk/fk mean "this IS a key." _id means "this HOLDS a key value."

Convention 02

Tables & Table Occurrences

This convention follows the Anchor-Buoy methodology. Base table names are clean, context-free PascalCase nouns. Table occurrence (TO) names encode the full relationship path from the anchor, making any TO self-describing in isolation.

Base Tables

PascalCase singular nouns. No prefixes, no context.

Base Table Examples
Customer       Invoice        Product
LineItem       Contact        SalesRep

Anchor TOs

Prefixed with a single underscore (_). This makes them immediately visually distinct in the Relationships panel.

Anchor TOs
_Customer      _Invoice
_Project       _PurchaseOrder

Buoy TOs

The anchor is always re-stated at the start, followed by the relationship path.

HopsPatternExample
Anchor_TableName_Customer
1 hop_Anchor__Table_Customer__Invoice
2 hops_Anchor__Table_Table_Customer__Invoice_LineItem
3 hops_Anchor__Table_Table_Table_Customer__Invoice_LineItem_Product
Role_Anchor__Table.Role_Customer__Contact.Billing
External_Anchor__@SourceTable_Reference__@MediaDocuments

Separator Reference

_ (prefix)
Marks an anchor TO — _Customer
__ (double)
Separates anchor from buoy chain — _Customer__Invoice
_ (internal)
Separates hops in the buoy chain — _Customer__Invoice_LineItem
.
Role disambiguation — _Customer__Contact.Billing
@
Marks external source + table as one token — _Reference__@MediaDocuments

Example Graphs

Customer Graph
_Customer
_Customer__Invoice
_Customer__Invoice_LineItem
_Customer__Invoice_LineItem_Product
_Customer__Contact.Primary
_Customer__Contact.Billing
_Customer__SalesRep
PurchaseOrder Graph (with external table)
_PurchaseOrder
_PurchaseOrder__Vendor
_PurchaseOrder__Vendor_Contact.Primary
_PurchaseOrder__LineItem
_PurchaseOrder__LineItem_Product
_PurchaseOrder__Receipt
_PurchaseOrder__Receipt_@WarehouseLineItem   ← external source
ℹ️
Quick Diagnostic Everything before __ tells you the anchor. Everything after, split on single _, gives you the relationship path. A name starting with _ and containing no __ is an anchor TO. A segment starting with @ is an external table.

Convention 03

Scripts

Core principle: Names tell you what and where. Headers tell you how. Parameters are JSON-keyed objects documented in the header block — never encoded in the name. All characters are URL-safe for use in fmp:// URLs without percent-encoding.

Script Name Structure

Anatomy
Module__VerbNoun
Module__VerbNoun--Qualifier
Module__VerbNoun--flag
Module__VerbNoun--Qualifier--flag

Startup__InitializeSystemVars--z         ← init flag · no qualifier
Startup__Login                            ← no flag · public entry point
Projects__CreateEditProject               ← no flag · public utility
Utility__ResetDataEntryFields--Loop--p    ← qualifier then private flag
Utility__GetRecordData--AsJSON             ← qualifier · no flag
Triggers__OnLayoutEnter                   ← trigger callback · no flag

Separator Roles

__ (double underscore)
Module boundary. Separates the module name from the script name. Appears exactly once per name.
-- (double hyphen)
Qualifier or flag boundary. Separates the core name from its qualifier, and the qualifier from its flag.
PascalCase
Word boundary within module, verb, noun, or qualifier. No separator character — words are distinguished by capitalisation.
⚠️
Formatting Rules PascalCase throughout — every word in every part of the name begins with a capital letter. The only exceptions are the single-letter flags (p, x, z), which are intentionally lowercase so they stand out as metadata. No spaces anywhere in the script name.

Suffix Flags

Flags appear as a suffix after --, following any qualifier. The absence of a flag is itself meaningful — it signals the script is a stable public utility safe to call from anywhere.

FlagNameCaller Rule
--pPrivateThis module only. Covers what was previously split into "private" and "subroutine" — any non-public helper gets --p. Must never be called from a button or from outside the module.
--zSetup / InitFile startup script or this module's own initialisation routine only.
--xDeprecatedExisting callers only. Header must note the replacement and target removal date. No new callers permitted.
(none)Public UtilityAnyone. Reusable across modules. Includes all user-facing entry points — if a button calls it, it has no flag.
ℹ️
Note on Sorting Unlike the previous convention, flags are now suffixes and do not affect alphabetical sort order. Organisation within a folder relies on consistent naming and the folder structure itself. Keep each module folder focused enough that scanning is manageable without sort tricks.

How to Choose a Flag

Ask: "Who is allowed to call this script?"

Anyone
Buttons, other modules, external scripts, fmp:// URLs → (none)
This module only
Internal helper, loop body, sub-process → --p
Startup only
File startup script or module init routine → --z
Nobody new
Kept only to avoid breaking existing callers → --x
💡
Trigger scripts (OnLayoutEnter, OnRecordLoad, etc.) live in a Triggers folder and get no flag. Their folder location signals that FileMaker's trigger system calls them — not developers directly.

Approved Verb List

All script verbs must come from this list. Never use a synonym and leave it undocumented. To add a new verb, update the standard first. All verbs are written in PascalCase as part of the script name.

Data Operations
Create  Update  Delete  Get      Set      Save
Validate Import  Export  Copy     Retrieve Select
Sort    Empty
UI & Navigation
Navigate Open    Close   Show    Hide    Toggle
Refresh  Resize  Maximize Rename Print   Send
Zoom
Process & Lifecycle
Initialize Process  Format  Reset     Clear  Undo
Handle     Evaluate Assign  Ping      Resequence
Capture    Mark     Perform
🚫
Deprecated — Do Not Use GO TO → use Navigate  |  Fetch / Load → use Get or Retrieve  |  Display → use Show  |  RUN → use Process or Perform  |  Unsort → use ClearSort  |  New with Create → redundant, drop it.

Qualifier Types

Qualifiers are appended after -- with no spaces, written in PascalCase. If a flag is also present, the qualifier comes first: Module__VerbNoun--Qualifier--flag.

TypeExamples
Context / Trigger--OnCreate  --OnEdit  --OnLogin  --EnterKey
Method / Approach--ByDirection  --ByNumber  --AsJSON  --UsingTemplate
Scope / Target--ServerSide  --PSoS  --Modal  --Batch
Condition--IfNotEmpty  --IfEmpty  --Loop
Version--v2  --Legacy
Combined OperationJoin directly in PascalCase — no qualifier needed: CreateEditProject, OpenInitializeWindow

Script Header Block

Header Template
# ─────────────────────────────────────────────────────────
# SCRIPT   : Invoice__CreateEditInvoice
# PURPOSE  : Opens the Invoice form. Creates a new record if
#            none exists, or edits the current record.
# ──────────────
# PARAMS   : invoiceID     | Text | ID of record to edit (empty = create new)
#            clientID      | Text | Pre-fill client on new record (optional)
#            returnLayout  | Text | Layout to return to on close
# ──────────────
# RETURNS  : $result = "success" | "error: <message>"
# ──────────────
# CALLED BY: Button – Invoice List > New Invoice
#            Button – Client Detail > Add Invoice
# ──────────────
# MODIFIED : 2026-03-23 | JD | Initial build
#            2026-04-01 | SR | Added returnLayout param
# ─────────────────────────────────────────────────────────

Naming Anti-Patterns

Anti-PatternBeforeAfter
Special characters / spaces[Startup] !! Initialize System VarsStartup__InitializeSystemVars--z
CREATE / EDIT slashCREATE / EDIT ProjectProjects__CreateEditProject
GO TOGO TO LayoutNav__NavigateToLayout
Bracket paramsRESET Fields [LOOP]Utility__ResetDataEntryFields--Loop--p
Noun-only namesDevDocsDev__OpenDevDocs
New + Create redundancyCreateNewRecordCreateRecord

URL Usage

All script names under this convention are unconditionally safe for use in fmp:// URLs without percent-encoding — no brackets, spaces, or special symbols appear anywhere in a compliant name.

fmp:// Example
fmp://$/DevHub?script=Utility__GetRecordData--AsJSON&param=...
💡
Exemptions The Scripts Menu folder is fully exempt — names there are user-facing and do not need to be URL-safe. Add-on scripts are exempt from module naming, but custom scripts added to an add-on folder follow the convention.

Convention 04

Layouts

Layout names follow a three-part structure that leads with the subject, describes what the layout does, and adds a qualifier only when needed. The result is a name that is self-describing in isolation and groups naturally when sorted.

Structure

Anatomy
Context | Purpose — Variant

Context  →  The primary table or subject area the layout serves
Purpose  →  What the layout does or what type it is
Variant  →  Optional qualifier when further distinction is needed

 |   pipe      separates Context from Purpose
 —   em dash   separates Purpose from Variant

Omit the Variant when no qualifier is needed.

General Principles

Lead with subject
Context comes first so related layouts group together when sorted alphabetically.
Describe purpose
Name what the layout does, not who uses it.
Spell things out
Avoid abbreviations — clarity over brevity.
Variant sparingly
Only add a Variant when it carries real meaning. Omit when the Purpose alone is sufficient.
Be consistent
Apply the same terms throughout the entire file. Don't mix "Detail" and "Form" for the same concept.

Standard Layouts

Everyday detail and list layouts use a plain Context | Purpose name.

Examples
Contacts | Detail
Contacts | List
Contacts | New Record
Invoices | Detail
Invoices | List
Products | Detail

Print Layouts

Use Print as the Purpose. Use the Variant to distinguish between different print formats for the same subject.

Examples
Contacts | Print — Label
Contacts | Print — Full Record
Contacts | Print — Summary
Invoices | Print — Standard
Invoices | Print — Detailed
Invoices | Print — Packing Slip

Card Windows

Always use Card as the Purpose. Use the Variant to describe the job the card performs, not how it is triggered.

Examples
Contacts | Card — Quick Edit
Contacts | Card — New Record
Contacts | Card — Merge Preview
Invoices | Card — Add Line Item
Invoices | Card — Send Confirmation
Products | Card — Picker
Products | Card — Detail

Portal Pickers

Use Picker as the Purpose, and name the Variant after what is being selected.

Examples
Invoices | Picker — Products
Invoices | Picker — Contacts
Projects | Picker — Employees
Projects | Picker — Tasks

Reports & Summaries

Use Report or Summary Report as the Purpose. The Variant can describe the grouping or filter applied.

Examples
Invoices | Summary Report — by Month
Invoices | Summary Report — by Sales Rep
Projects | Report — Status Overview
Contacts | Report — Mailing List

Utility & Internal Layouts

Prefix layouts not meant for end-user navigation with an underscore (_). This pushes them to the top of sorted lists and signals internal use.

Examples
_Globals
_Globals | Card — User Preferences
_Globals | Card — Navigation Menu
_Email Template

Deprecated Layouts

Prefix retired layouts with z_ so they sort to the bottom and are clearly marked as inactive.

Examples
z_Deprecated | Contacts — Old Layout
z_Deprecated | Invoices — Legacy Print

Convention 05

Layout Objects

Object names follow a BEM-adapted methodology. Because FileMaker requires all object names to be unique (like CSS IDs, not classes), every name must be fully descriptive in isolation.

Core Syntax

BEM Structure
block__element--modifier

block    →  Parent container or logical group  (sidebar, modal, record-card)
element  →  Component living inside the block  (title, divider, button)
modifier →  Variant, state, or disambiguator   (active, primary, 1)

All segments lowercase. Words within a segment separated by a single hyphen.

✓  nav-bar__menu-item--active
✗  NavBar__MenuItem--Active
✗  nav_bar__menu_item--active

Blocks

A block is a standalone, self-contained section of a layout. Name objects belonging to a block using that block name as the prefix. Do not nest block names — if nesting seems necessary, consider giving the inner group its own block name.

Elements

An element belongs to and only makes sense inside its block. Use names that describe purpose or role, not appearance.

💡
Prefer sidebar__divider (describes role) over sidebar__gray-line (describes appearance). Appearance-based names break when the design changes.

Semantic vs. Instance Modifiers

When multiple identical objects share the same block and element name, choose between a semantic modifier or a numeric instance modifier.

Semantic
Objects differ in purpose, position, or behavior → sidebar__divider--top, sidebar__divider--footer
Instance
Objects are functionally identical → sidebar__divider--1, sidebar__divider--2

Do not mix semantic and instance modifiers on the same element type within a block. Choose one approach and apply it consistently.

Quick Reference

sidebar
Block — standalone container or group
sidebar--collapsed
Block with modifier — variant of the block itself
sidebar__title
Block + element — belongs to the sidebar block
sidebar__divider--top
Block + element + semantic modifier
sidebar__divider--1
Block + element + instance modifier (identical objects)
overlay
Standalone non-grouped object

Full Example — Sidebar

Sidebar Layout Group
sidebar                        ← container/group object
sidebar__title                 ← text: group heading
sidebar__divider--1            ← shape: first divider (no semantic distinction)
sidebar__divider--2            ← shape: second divider
sidebar__nav-item--reports     ← button: navigates to reports
sidebar__nav-item--dashboard   ← button: navigates to dashboard
sidebar__nav-item--settings    ← button: navigates to settings
sidebar__footer-label          ← text: version or user info

Rules Summary

1. All names lowercase with hyphens as word separators. 2. Double underscores separate block from element. 3. Double hyphens separate element from modifier. 4. Do not exceed two levels (block__element). 5. Prefer semantic modifiers when a meaningful distinction exists. 6. Use instance modifiers only for functionally identical objects. 7. Document block vocabulary in a reference sheet to keep names consistent across layouts.

Convention 06

Theme Styles

Theme style names adapt BEM principles for FileMaker's Inspector panel. Every name should communicate what it is, where or how it is used, and any variant or state — at a glance.

Structure & Separators

Syntax
block__element--modifier

-    single hyphen    →  joins words within a segment
__   double underscore →  separates block from element
--   double hyphen    →  separates block/element from modifier

prt__leading-grand-summary--brand
 │            │                  └── modifier
 │            └── element (multi-word, joined by -)
 └── block

Naming Principles

1. Name for role, not appearance. Prefer --muted over --gray, and --brand over --blue, so names survive a color theme change.

2. Keep block names short. txt, btn, prt are easy to scan. Avoid full words where the abbreviation is unambiguous in your system.

3. Depth limit: block__element--modifier only. Do not go beyond three segments. prt__leading-grand-summary--brand is acceptable. prt__header__label--brand--active is too deep.

4. One modifier per -- separator. Flatten compound modifiers with a single hyphen: --ruled-alt, not --ruled--alt.

5. Document your vocabulary. Maintain a reference list of approved blocks, elements, and modifiers so the convention stays consistent as the theme grows.

Standard 07

Layout Sizing Standards

FileMaker canvas dimensions and all measurements are expressed in points (pt) — the unit used in FileMaker's Inspector panel when set to Points mode. These standards are adapted from Apple Human Interface Guidelines and Material Design touch target rules, converted to FileMaker's point-based unit system. Desktop and tablet layouts use landscape orientation; phone layouts use portrait.

ℹ️
pt ≠ px 1pt = 1.333px. To convert web/RWD pixel values to FileMaker points: pt = px × 0.75. The most important conversion: the standard 44px touch target minimum becomes 33pt in FileMaker.

FileMaker Go does not reflow layouts — canvas size is your breakpoint decision. Design three separate layout families and route users at startup.

Touch Target Padding Even if a button visually appears smaller than 33pt tall, its clickable bounds should be padded to meet the 33pt minimum. This is especially important for icon-only buttons and nav bar items in FileMaker Go.

Layout-Specific Conventions

Desktop — Persistent left sidebar or top tab bar navigation. Card windows are appropriate for modals, pickers, and sub-forms. Two- and three-column portal/list views are suitable.

Tablet (Landscape) — Treat as a constrained desktop. Top nav bar or slim sidebar works well. Avoid small controls — fingers are in play even with a keyboard attached. Card windows should stay no wider than ~80% of canvas.

Phone (Portrait) — Single-column layouts only. No horizontal-scrolling portals. Bottom-anchored nav bars feel native in FileMaker Go. Use popover buttons and collapsible sections for secondary content.

Device Detection Script

Use at startup to route users to the appropriate layout family based on platform and window width.

Startup — Route to Device Layout
// Startup — Route to Device Layout
If [ Get(SystemPlatform) = 3 or Get(SystemPlatform) = -3 ]
    // iOS (FileMaker Go)
    If [ Get(WindowWidth) >= 700 ]
        // Tablet — landscape
        Go to Layout [ "Home [ tablet ]" ]
    Else
        // Phone — portrait
        Go to Layout [ "Home [ phone ]" ]
    End If
Else
    // Desktop (Mac or Windows)
    Go to Layout [ "Home [ desktop ]" ]
End If
